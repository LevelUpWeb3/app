# **Why is TLSNotary Important?**

## **The Limitations of TLS for Non-Repudiation**

[Transport Layer Security (TLS)](https://en.wikipedia.org/wiki/Transport_Layer_Security)Â plays an important role in digital security by safeguarding communications between users and servers. It provides:

- **Data Integrity**: Ensures data is not altered during transmission.
- **Server Authentication**: Validates the serverâ€™s identity using Certificate Authorities (CAs).

However, while TLS is effective in protecting data transmitted between user and server, it falls short in providingÂ **non-repudiation**Â â€” the ability to prove to a third party that data is authentic and untampered.

This limitation arises because TLS uses a keyed hash (Message Authentication Code, or MAC), which requires the user (e.g., Alice) to know the key. This means Alice could potentially alter the data and recompute the hash after the TLS session, making it impossible to verify the dataâ€™s authenticity to another party (e.g., Bob).

**The Solution?**Â Digital signatures based on asymmetric public-key cryptography are required. Unfortunately, TLS does not natively support server-side signing of data, and even if it did, such an approach might compromise privacy if not designed thoughtfully. # **How TLSNotary Works**

The TLSNotary protocol operates through three main steps:

1. **Multi-Party TLS Request:**Â Traditional TLS connections are between a user and a server, but TLSNotary introduces a third participant: theÂ **Verifier**. Unlike a â€œ[man-in-the-middle](https://en.wikipedia.org/wiki/Man-in-the-middle_attack),â€ the Verifier operates within a secureÂ **Multi-Party Computation (MPC)**Â setup, ensuring that user data is not exposed in plaintext. To the server, the connection appears as a standard TLS request, preserving its integrity.*Alice and Bob can jointly manage the TLS connection, ensuring Bob can independently verify the dataâ€™s authenticity without seeing unshared portions.*
2. **Selective Disclosure:**Â TLSNotary allows the Prover toÂ **selectively disclose specific portions**Â of the data to the Verifier. Sensitive information can be redacted before sharing.Â ****Zero-Knowledge Proofs can be used to verify properties of redacted data without revealing the actual content.*Alice can choose which parts of the data to disclose, safeguarding her privacy.*
3. **Data Verification:**Â The Verifier can then authenticate the disclosed data by validating the server certificate using trusted Certificate Authorities (CAs). This ensures that non-redacted portions of the data match their cryptographic proofs.*With this approach, Alice retains full control over her data while providing Bob with cryptographic proof of its authenticity.*
